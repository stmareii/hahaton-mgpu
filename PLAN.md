# План анализа и очистки данных хакатона МГПУ

## Контекст

**Датасет:** `data/hakaton.csv` - 25 628 записей, 21 столбец, разделитель `;`.

**Цель:** выявить нарушения частоты тестирования (>1 раза в 3 мес.), аномалии и паттерны,
затем подготовить чистые данные для обучения модели (детектор аномалий / классификатор).

## Этап 1 - Базовый EDA и инвентаризация качества

### 1.1 Типы данных и пропуски
- Привести типы:

    `bdate`, `guard_bdate`,`test_date` ---> `datetime64`;
    
    `class` ---> `int`;
    
    `result` ---> `category`
- Подсчитать `null`/пустые строки по каждому столбцу
- Отдельно: `middle_name`, `guard_middle_name` - ожидаемо пустые (отчество необязательно)

### 1.2 Нормализация `result`
Текущее состояние - 8 вариантов написания двух значений + одна строка с датой вместо значения.
```
result.str.strip().str.lower() ---> {'достаточный', 'недостаточный'}
```
Строка с датой `2025-05-15` ---> `NaN` или отдельная метка `INVALID_RESULT`.

### 1.3 Числовые и форматные аномалии в `id_doc` / `guard_id_doc`
- Отрицательные значения (`-1236233`)
- Значение `0000000` - заглушка
- Буквенно-числовые (`N18005999`, `-N0849879`) - возможно, другой формат документа
- Создать булев флаг `id_doc_suspicious` для всех нестандартных значений

### 1.4 Распределения ключевых полей
- `class`: ожидается 1–11, проверить выбросы за пределами
- `bdate` / `guard_bdate`: проверить реалистичность возраста
  - ребёнок: 2010–2019 ---> возраст 7-16 лет (2026)
  - родитель: должен быть старше ребёнка минимум на 14 лет
- `test_date`: диапазон 2025-06 – 2026-02, проверить будущие/прошлые выбросы
- `variant`: числовое поле, распределение вариантов

## Этап 2 - Идентификация ребёнка и нарушения частоты

### 2.1 Составной ключ идентификации
Идентификатор ребёнка - сочетание:
```
(last_name, first_name, middle_name_norm, bdate)
```
`middle_name_norm` = `NaN` --> пустая строка (для устойчивости).

**Альтернативный ключ**: если `id_doc` не подозрительный - использовать его как первичный и проверять совпадение с ФИО+ДР.

### 2.2 Нарушения 3-месячного интервала
```python
df.sort_values(['child_id', 'test_date'])
df['prev_test_date'] = df.groupby('child_id')['test_date'].shift(1)
df['days_since_last'] = (df['test_date'] - df['prev_test_date']).dt.days
df['frequency_violation'] = df['days_since_last'] < 90
```

Метрики для дашборда:
- % детей с хотя бы одним нарушением
- Топ-школы по количеству нарушений (`ogrn_naprav`)
- Временной тренд нарушений по месяцам

### 2.3 Один ребёнок - разные документы
Найти случаи, когда `(last_name, first_name, bdate)` совпадает, но `id_doc` различается - вероятная ошибка ввода или смена документа.

## Этап 3 - Аномалии в данных

### 3.1 Логические противоречия
| Проверка | Условие |
|---|---|
| Ребёнок старше/ровесник родителя | `bdate <= guard_bdate` или разница < 14 лет |
| Класс не соответствует возрасту | `class` vs `bdate`: класс ≈ год - 2026 + 7 ± 2 |
| Пол ребёнка vs имя | Проверка по словарю типичных мужских/женских имён |
| Родитель = ребёнок | ФИО и ДР родителя совпадают с ребёнком |
| `test_date` < `bdate` | Тест раньше рождения |
| Дубликаты `our_number` | Уникальный номер результата |

### 3.2 Дубликаты записей
- Полные дубликаты (все поля совпадают)
- Частичные: один `our_number` с разными результатами

### 3.3 Аномалии на уровне школ
- Школы с нереально высоким числом тестирований в день
- Школы, где все дети одного класса тестировались в один день (`test_date` + `class` + `ogrn_area`)
- Школы с аномально высоким % нарушений частоты

### 3.4 Аномалии в `variant`
- Один ребёнок прошёл один и тот же вариант дважды (возможный повтор для улучшения)
- Редкие варианты (< N записей) - возможные тестовые/технические записи

## Этап 4 - Очистка и заполнение пропусков

### 4.1 Что удаляем (жёсткие правила)
- Полные дубликаты ---> `drop_duplicates`
- Записи с `result = INVALID_RESULT` (дата в поле результата) - удалить или вынести в отдельный датафрейм для отчёта
- Записи с `test_date < bdate` - физически невозможны

### 4.2 Что помечаем флагом (не удаляем)
```python
anomaly_flags = {
    'frequency_violation',     # нарушение 3-месячного правила
    'id_doc_suspicious',       # невалидный id_doc
    'age_class_mismatch',      # класс не соответствует возрасту
    'parent_age_anomaly',      # родитель слишком молод
    'duplicate_our_number',    # дублирующий номер результата
    'same_day_mass_test',      # массовое тестирование в один день
}
```
Для каждой записи создаём столбец `anomaly_score` (сумма флагов) и `anomaly_labels` (список сработавших флагов).

### 4.3 Заполнение пропусков

| Поле | Стратегия |
|---|---|
| `middle_name`, `guard_middle_name` | Оставить `NaN` - это валидное отсутствие; при векторизации ---> пустая строка |
| `id_doc` подозрительные | Не заполнять; использовать флаг `id_doc_suspicious` как признак |
| `class` (если пусто) | Вывести из `bdate` и `test_date` по формуле возраста ± 1 класс |
| `gender` (если пусто) | Вывести из имени по словарю (или простой ML-классификатор на именах) |

## Этап 5 - Подготовка признаков для модели

### 5.1 Числовые признаки
- `age_at_test` = `(test_date - bdate).dt.days / 365.25`
- `parent_age_at_test` = аналогично
- `parent_child_age_diff` = `parent_age_at_test - age_at_test`
- `days_since_last_test` (из этапа 2.2, `NaN` для первого теста)
- `tests_count_child` = кол-во тестов данного ребёнка за всё время
- `school_daily_test_count` = кол-во тестов в данной школе в данный день

### 5.2 Категориальные признаки (кодировка)
- `gender`, `guard_gender` ---> 0/1
- `result` ---> 0 (недостаточный) / 1 (достаточный)
- `class` ---> int
- `ogrn_naprav`, `ogrn_area` ---> target encoding (по % нарушений) или frequency encoding

### 5.3 Текстовые признаки (опционально)
- `name_naprav`, `name_area` - не использовать напрямую; только через ОГРН

### 5.4 Целевая переменная для модели аномалий
**Вариант A** (unsupervised): использовать `anomaly_score` как метку для Isolation Forest / DBSCAN.  
**Вариант B** (supervised): `is_anomaly = anomaly_score > 0` ---> бинарная классификация (LightGBM/XGBoost).

## Этап 6 - Модель и дашборд

### 6.1 Модель аномалий
- **Рекомендуемый базовый подход**: Isolation Forest (`sklearn.ensemble.IsolationForest`) на числовых признаках из п.5.1
- **Улучшение**: LightGBM на признаках из п.5.1–5.2 с таргетом из п.5.4 вариант B
- Метрики: precision/recall на флагах из этапа 3

### 6.2 Дашборд (Streamlit или Plotly Dash)
Страницы:
1. **Обзор**: kpi-карточки (всего записей, % нарушений, % аномалий)
2. **Нарушения частоты**: таблица + карта по школам
3. **Аномалии**: таблица с фильтрами по типу флага
4. **Топ школ**: рейтинг по нарушениям и аномалиям
5. **Модель**: ввод записи вручную ---> предсказание аномалии
